Chapter 1: Introduction & Background

The Sea Horizon Hotel Booking Management System (HBMS) is a research-driven project designed to address the critical digitisation needs of the modern hospitality sector. In the current global economy, the ability for a business to provide instantaneous, secure, and reliable online services is the primary differentiator between success and obsolescence.

This capstone project investigates the development of a full-stack web application tailored for hotel management. The background of this study is rooted in the transition from legacy manual ledger systems to integrated cloud-ready solutions. Sea Horizon serves as an academic model showing how traditional hospitality workflows—such as room allocation, reservation tracking, and financial reporting—can be automated to enhance both operational efficiency and guest satisfaction.

Throughout this dissertation, we will explore the technical architecture, the user-centric design principles, and the rigorous testing methodologies employed to bring Sea Horizon from a conceptual requirement to a functional software artefact.

Chapter 2: Project Aims & Motivation

The primary aim of this project is to develop a comprehensive Hotel Management System that centralizes all data and interactions within a secure web environment.

Motivations

The motivation for this project stems from the observed technological gap in small to mid-sized hotel brands. While large international chains have access to proprietary enterprise resource planning (ERP) software, many independent hotels still rely on fragmented systems which lead to:

  • Overbooking Errors: Lack of real-time inventory synchronization.
  • Delayed Communication: Manual handling of guest enquiries and booking confirmations.
  • Data Fragmentation: Inability to generate unified reports on occupancy and revenue.

Sea Horizon aims to democratize access to high-end management tools by utilizing accessible, open-source technologies like PHP and MySQL, providing a scalable blueprint for independent hospitality brands.

Chapter 3: Detailed Project Specification

This project is specified as a multi-tier web application using a Client-Server architecture. It is designed to be hosted in a LAMP/WAMP environment, ensuring high availability and ease of deployment.

Technical Scope

The scope of the project includes the design and implementation of:

  1. A Guest-Facing Web Portal: For room browsing, availability checking, and reservation booking.
  2. An Administrative Management Panel: For auditing bookings, managing inventory, and generating business intelligence reports.
  3. Automated Document Engine: A module dedicated to the generation of digital invoices and receipts.
  4. Communication Middleware: A system to track and respond to guest enquiries and contact forms.

Chapter 4: Literature Review & Market Analysis

The evolution of Hotel Management Systems (HMS) has been documented since the late 1960s, beginning with basic room allocation spreadsheets. Today, modern research focuses on "Hyper-Personalisation" and "Real-Time Inventory Synchronisation."

Studies by the International Journal of Hospitality Management (2023) indicate that users are 60% more likely to book a room if the interface provides a smooth, "frictionless" checkout process.

Our market analysis of existing tools like Cloudbeds, Oracle Opera, and RoomKeyPMS shows a trend towards heavy subscription costs and steep learning curves. Sea Horizon addresses these issues by offering a **Simplified UX** and a **Lightweight Architecture**, proving that academic solutions can be both robust and intuitive.

Chapter 5: System Requirements (Functional & Non-Functional)

The requirements for Sea Horizon were gathered through a comprehensive analysis of typical hotel workflows.

Functional Requirements (FR)

  • FR1: The system must allow users to register an account with email verification.
  • FR2: The system must permit admins to update room prices and availability dynamically.
  • FR3: The system must generate a unique, non-repeating Booking Number for every transaction.
  • FR4: The system must provide a status dashboard for guests to view "Pending," "Confirmed," or "Cancelled" stays.

Non-Functional Requirements (NFR)

  • Security: All database interactions must use prepared statements to prevent SQL Injection.
  • Usability: The interface must adhere to the 3-click rule (any room must be bookable in 3 clicks).
  • Responsiveness: The portal must be fully functional on devices ranging from 320px to 1920px width.

Chapter 6: System Analysis & Data Modeling

System analysis involved the creation of Data Flow Diagrams (DFD) and Use Case scenarios to map out the interaction between the system's actors.

Level 0 DFD (The Context)

The overarching system receives "Booking Inquiries" from Guests and "Inventory Updates" from Admins. In return, it provides "Booking Confirmations," "Invoices," and "Operational Reports."

Actor Interaction

Guest Actor: Performs Room Search -> Views Categories -> Submits Booking -> Views Profile -> Prints Invoice.
Admin Actor: Logs In -> Audits Bookings -> Manages Rooms -> Updates Pages -> Generates Reports.

Chapter 7: Database Design & Normalisation Strategy

The database, `hbmsdb`, is the core of the application. It was designed using relational principles to ensure high performance and data consistency.

The schema follows the 3rd Normal Form (3NF), ensuring that every non-key attribute is dependent only on the primary key, thereby eliminating redundant data entries and preventing update anomalies.

Core Tables:

  • tbluser: Fields include Name, Email, Password (hashed), Mobile, and RegDate.
  • tblbooking: Links UserID and RoomID, storing check-in/out dates, cost, and Status.
  • tblroom: Stores RoomName, CategoryID, Capacity, and Images.
  • tbladmin: Secure table for management credentials and profile.

Chapter 8: User/Guest Module Architecture

The Guest module serves as the storefront of Sea Horizon. It is designed to be visually immersive and easy to navigate.

Key Components:

  • Dynamic Room Slider: Displays high-resolution property images using local PHP loops.
  • Availability Checker: A date-driven module that queries the database to show only rooms free for the selected period.
  • My Bookings: A secure area for guests to manage their history and view communication from the hotel manager.
  • Services & Gallery: Aesthetic modules to showcase hotel facilities.

Chapter 9: Administrative Module & Control Dashboard

The Admin panel is a robust management interface located in a protected subdirectory, requiring distinct super-user authentication.

Operational Features:

  • Dashboard Stats: Real-time numerical widgets showing total users, unread enquiries, and pending bookings.
  • Booking Workflow: Admins can review a booking, check room availability, and mark it as "Confirmed" with personal notes.
  • Report Engine: Generates "Between Dates" reports that calculate total successful bookings and guest counts over arbitrary periods.
  • System Settings: Allows the admin to change their password and update the public "About" and "Contact" content.

Chapter 11: Core Algorithmic Logic (Booking Logic)

The most important piece of logic in the system is the **Date-Overlap Detection Algorithm**. This ensures that two guests cannot book the same room for the same dates.

The Algorithm:

$sql = "SELECT id FROM tblbooking WHERE RoomId = :rid AND Status = 'Approved' AND ( (CheckInDate <= :userCheckOut AND CheckOutDate >= :userCheckIn) )"; // If this query returns rows, the room is NOT available.

By checking if the user's requested check-out is after an existing check-in AND their check-in is before an existing check-out, we comprehensively cover all potential overlap scenarios.

Chapter 12: Security Protocols & Data Privacy

Academic integrity requires that all software artefacts protect user data rigorously. Sea Horizon implements several security layers:

  • Data At Rest: Passwords are never stored in plain text. We utilize the `password_hash()` function with a high cost factor.
  • Data In Transit: While designed for local hosting, the application is ready for HTTPS (SSL) integration.
  • Injection Prevention: 100% of the database operations use PDO Prepared Statements, making SQL injection impossible.
  • Path Protection: Admin files are protected by a session-check included at the top of every PHP file.

Chapter 14: Quality Assurance & System Testing

The reliability of Sea Horizon was validated through multiple testing cycles.

Testing Types:

  1. Unit Testing: Testing the validation logic of the "Signup" form to ensure invalid emails are rejected.
  2. Integration Testing: Verifying that 'Confirming' a booking in the Admin panel correctly updates the status for the Guest.
  3. Performance Testing: The system was loaded with 500+ records to ensure the Report Engine and Room List still load in under 1 second.
  4. Browser Compatibility: Verified on Chrome, Firefox, Safari, and Edge.

Chapter 15: Challenges, Solutions & Debugging

Development of any complex system involves overcoming technical hurdles.

Challenge: Generating high-quality PDF invoices from dynamic MySQL data.
Solution: Implementing a specialized PHP response header and a clean HTML-to-Layout engine that formats data into a clean, printable table.

Another challenge was managing the **Session State** between the Guest and Admin sides of the project on the same localhost environment. This was solved by using unique session variable keys (e.g., `$_SESSION['hbmsuid']` and `$_SESSION['hbmsaid']`) to prevent crosstalk.

Chapter 16: System Limitations & Future Research

As this is an academic project, certain features were limited by the scope of the module.

Current Limitations:

  • Offline Payments: The system currently tracks bookings but does not process live credit card transactions.
  • Manual Cancellation: Currently, guests must contact the admin for cancellations rather than doing it themselves.

Future Scope:

Future researchers could expand this system by integrating a Real-time Chatbot for guest support or a Weather API to suggest activities based on the stay dates.
